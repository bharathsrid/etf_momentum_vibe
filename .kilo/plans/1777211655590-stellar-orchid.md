# Plan: Multi-Model Configuration for Kilo (Global)

## Goal
Configure Kilo globally to use:
- **GLM 5.1** (`openrouter/z-ai/glm-5.1`) as the orchestrator (primary agent)
- **GLM 5.1** (`openrouter/z-ai/glm-5.1`) for planning subtasks
- **MiniMax 2.7** (`openrouter/minimax-2.7`) for execution subtasks

## Files to Create (all under `~/.config/kilo/`)

### 1. `~/.config/kilo/kilo.json` — Default model config

```jsonc
{
  "$schema": "https://app.kilo.ai/config.json",
  "model": "openrouter/z-ai/glm-5.1",
  "small_model": "openrouter/z-ai/glm-5.1",
  "default_agent": "orchestrator"
}
```

Sets GLM 5.1 as default for orchestrator and summaries. `default_agent` points to the orchestrator agent.

### 2. `~/.config/kilo/agent/orchestrator.md` — Primary orchestrator (GLM 5.1)

```markdown
---
description: Main orchestrator that plans and delegates work to subagents
mode: primary
model: openrouter/z-ai/glm-5.1
steps: 50
permission:
  bash: allow
  edit: allow
  read: allow
  task: allow
---
You are the orchestrator. You plan tasks, break them into subtasks, and delegate execution to the executor subagent.

CRITICAL RULES:
- You MUST use the Task tool with subagent_type "executor" for ALL code implementation tasks. NEVER write or edit code yourself.
- You MUST use the Task tool with subagent_type "planner" when detailed codebase analysis or planning is needed.
- You may think, analyze, and coordinate yourself, but all code changes go through the executor.
- After receiving executor results, review them for correctness and coordinate next steps.

When delegating to the executor, provide clear, specific instructions about what to implement, which files to modify, and any conventions to follow.
```

### 3. `~/.config/kilo/agent/planner.md` — Planning subagent (GLM 5.1)

```markdown
---
description: Analyzes requirements and creates detailed implementation plans
mode: subagent
model: openrouter/z-ai/glm-5.1
steps: 15
hidden: false
permission:
  read: allow
  grep: allow
  glob: allow
---
You are the planner. You analyze the codebase and requirements, then produce detailed implementation plans. You do NOT write code — you produce plans with clear steps. Focus on understanding the existing codebase patterns and conventions before planning.
```

### 4. `~/.config/kilo/agent/executor.md` — Execution subagent (MiniMax 2.7)

```markdown
---
description: Executes coding tasks delegated by the orchestrator
mode: subagent
model: openrouter/minimax-2.7
steps: 25
hidden: false
permission:
  bash: allow
  edit: allow
  read: allow
---
You are the executor. You receive specific coding tasks from the orchestrator and implement them. Focus on writing clean, correct code. Follow the instructions precisely and return your results.
```

### 5. `~/.config/kilo/command/plan.md` — Slash command for planning

```markdown
---
description: Analyze and plan using GLM 5.1
agent: planner
model: openrouter/z-ai/glm-5.1
---
Analyze the following request and create a detailed implementation plan: $ARGUMENTS
```

### 6. `~/.config/kilo/command/execute.md` — Slash command for execution

```markdown
---
description: Execute a task using MiniMax 2.7
agent: executor
model: openrouter/minimax-2.7
subtask: true
---
Implement the following task: $ARGUMENTS
```

## Workflow

**Use the orchestrator as your primary agent** (select via `/agents` or it's auto-selected as `default_agent`).

1. You interact with the **orchestrator** (GLM 5.1) as your primary agent
2. For planning, the orchestrator delegates to the **planner** subagent (GLM 5.1) via Task tool with `subagent_type: "planner"`
3. For execution, the orchestrator delegates to the **executor** subagent (MiniMax 2.7) via Task tool with `subagent_type: "executor"`
4. The orchestrator reviews executor output and coordinates next steps

### One-shot alternatives (optional)
- `/plan <request>` — Run planner directly for a single planning task
- `/execute <task>` — Run executor directly for a single execution task

### Important caveat
The orchestrator is an LLM following a system prompt — it *should* delegate execution to the executor, but nothing programmatically prevents it from writing code itself. If you notice it implementing code directly instead of delegating, remind it to "delegate to the executor".

## Runtime Controls

- Switch agents: `<leader>a` or `/agents`
- Switch models: `<leader>m` or `/models`
- Cycle agents: `tab` / `shift+tab`

## Implementation Steps

1. Create `~/.config/kilo/kilo.json` with default model config
2. Create `~/.config/kilo/agent/` directory
3. Create `orchestrator.md`, `planner.md`, `executor.md` agent files
4. Create `~/.config/kilo/command/` directory
5. Create `plan.md`, `execute.md` command files
6. Verify by running `/agents` and `/models` in Kilo
